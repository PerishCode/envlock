# v0.4.7

- Keep Yarn classic global package contents and executable bins inside envlock-managed paths, so `yarn global add` works in the same envlock-owned Node ecosystem as npm and pnpm.
- Preserve package-manager-aware `pnpm` and `yarn` routing under corepack and project-level `packageManager`, while keeping the selected Node version authoritative.
- Continue converging the node toolchain flow toward fully envlock-owned local dogfooding, without depending on host-managed Node managers such as `fnm`.

## Validation

- Published-artifact install validation against `v0.4.7-beta.1` on GNU libc Linux and local macOS temp sandboxes.
- Pure envlock-owned local rehearsal using an isolated toolchain under `~/.envlock/toolchains/node/v24.14.0`, with fresh temp `HOME`, `ENVLOCK_HOME`, `ENVLOCK_LOG_HOME`, project, and state directories.
- Concrete command sets included:
  - `envlock plugin node init`
  - `envlock plugin node apply --node-bin ... --npm-bin ... --pnpm-bin ... --yarn-bin ...`
  - `npm install`
  - `pnpm add`
  - `npm install -g cowsay`
  - `pnpm add -g rimraf`
  - `yarn global add http-server`
- Result: `node`, `npm`, `pnpm`, `yarn`, and their installed global binaries resolved through envlock-managed paths; package and global installs succeeded in a fully isolated local rehearsal without touching the live `fnm` setup.
