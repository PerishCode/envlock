# 设计边界

`envlock` 的边界是刻意收窄的。

## 擅长做什么

- 从 JSON 构建确定性的环境会话。
- 组合静态值与简单环境变量操作。
- 通过 `command` 注入桥接动态引导工具。
- 将配置应用到 shell 输出或进程级 command mode。

## 刻意不做什么

- 不自动持久化全局 shell 变更。
- 不直接管理包版本或运行时安装器。
- 不替代完整任务运行器或 shell 框架管理器。

## 为什么要这样划界

窄边界让行为更可审计：

- profile 明确且可版本化。
- 输出可在应用前先审阅。
- 运行时在 CI 与本地 shell 中都更可预测。

因此，复杂路由逻辑（例如工具级 Agent 选择）更适合放在调用方脚本或 alias，`envlock` 只专注环境组装。
